# logzilla-studio-next — редизайн «Inspector Split»

Форк `logzilla-studio`. Цель: перенести визуальный язык **Concept 11** (3-pane
«режим расследования», фиолетовая тема, Inter Tight + JetBrains Mono) в рабочее
приложение — **не ломая контракт и стек ядра**.

> Оценка и решения ниже даны в четырёх ролях, как было запрошено: архитектор,
> UX-дизайнер, исследователь, консультант-аналитик. Это рамка для разговора,
> а не разные авторы.

---

## 1. Что было на входе

| Артефакт | Что это | Пригодность |
|---|---|---|
| `logzilla-studio/` | Рабочее приложение: Python stdlib HTTP-сервер + vanilla JS, граница UI↔ядро на HTTP/JSON. Двухколоночный лейаут (загрузка+настройки / предпросмотр). | Прод-основа. |
| `concept-11-source/` | Статический мокап на React+Babel из CDN, хардкод `SAMPLE_LOGS`, без бэкенда. 3 колонки. | Визуальный язык, не код. |

**Главное наблюдение:** концепт и приложение живут на **разных стеках**. Концепт
тащит React/Babel-standalone (компиляция JSX в браузере) только чтобы нарисовать
статику. Приложение принципиально zero-dependency.

---

## 2. Архитектор — ключевое решение по стеку

**Перенёс дизайн концепта на vanilla JS/CSS, React/Babel из концепта НЕ тащу.**

Почему:

- **Граница ценна, стек концепта — нет.** Ценность studio — в контракте
  `/api/parse · /api/export` и в наблюдаемости (`observability.js`, `/api/client-log`,
  структурные логи с `run_id`). Концепт не добавляет к этому ничего; он добавляет
  только внешний вид. Внешний вид переносится CSS-переменными и сеткой.
- **Babel-in-browser — это не прод.** Компиляция JSX на каждой загрузке — то, от
  чего сам README концепта предупреждает («для прод-сборки JSX лучше
  прекомпилировать»). Тянуть в рабочее приложение CDN-зависимости и шаг сборки
  ради косметики — отрицательный размен.
- **Отлаживаемость ИИ сохранена.** Решение проекта — держать границу на самом
  отлаживаемом слое (HTTP + DevTools). Vanilla DOM это сохраняет; React-дерево с
  in-browser-трансформом её бы только размыло.

Бэкенд (`app/`, контракт, тесты) перенесён **без единого изменения**. Менялся
только слой представления `web/`.

```
web/index.html   — 3-pane разметка + маскот инлайн-SVG (без React)
web/styles.css   — палитра/типографика Concept 11 на CSS-переменных
web/core.js      — ЧИСТЫЙ слой: извлечение полей (level/source/req_id/ts/msg),
                   разбор поиска (lvl:/src:/req:), подсветка. Без DOM.
web/app.js       — слой DOM/IO: тот же контракт, рендер потока/инспектора, выбор
web/observability.js — без изменений (слой наблюдаемости)
tests/core.test.mjs — юнит-тесты чистого слоя (node:assert, 0 зависимостей)
```

**Разделение core/DOM — осознанный senior-выбор.** Самые хитрые места
(`sourceOf`, `reqIdOf`, `buildQuery`) уже дважды давали баги (длина req_id,
HTTP-глагол как источник). Вынесены в `core.js` — модуль без DOM, грузится и в
браузере (глобал `LZ`), и в Node (`require`) **без шага сборки** (UMD-обёртка).
Это позволило покрыть их юнит-тестами:

```bash
node --test 'tests/*.test.mjs'        # 7 наборов, 0 npm-зависимостей
```

Рендер строит сообщения и подсветку через DOM-узлы (`textContent` + `<mark>`), а не
`innerHTML` → XSS невозможен по построению. Выделение перекрашивается на месте
(`paintSelection`), без полного ре-рендера потока на каждый клик/стрелку.

## 3. Исследователь — маппинг концепта на реальность

Концепт рисует то, чего у ядра **пока нет**. Честный разбор, чтобы не продать
картинку за функциональность:

| Элемент концепта | Реальная опора | Решение |
|---|---|---|
| Дерево live-файлов сессии с размерами | ~~studio парсит **один** inline-источник за раз~~ → **реализовано** (см. ниже) | Левая колонка = **дерево файлов сессии** (имя · размер · формат · счётчик ошибок) + настройки парсера. Drop/paste **добавляет** файл; каждый = независимый `ParseRequest` через тот же single-source контракт. Активный файл питает поток/инспектор. |
| Сохранённые фильтры со счётчиками | Есть только фильтр по уровням | Уровни — **кликабельные плитки** с цветным фоном и живыми счётчиками. Клик гасит уровень в потоке мгновенно (клиентский фильтр) и уходит в `options.log_levels` при следующем «Парсить». |
| Поток `# ts lvl source message` | `records[]` с произвольной схемой (text→`raw`, JSON/CSV→поля) | LVL — залитый цветной бейдж. SOURCE: структурное поле, а для plain-text — **вызываемый компонент**, извлечённый из строки (токен после уровня; HTTP-глаголы/пути отброшены). `api-gateway`, `cache`, `worker-3` вместо «—». |
| Inspector: JSON, Контекст, Похожие | Запись + связанные строки окна | Структура (pretty-JSON записи), **Контекст — все строки того же `req_id`/`trace_id`** (структурное поле или вытяжка из текста; берётся из полного окна, мимо фильтров, чтобы не рвать трассу; откат на ±6 соседей, если id нет), Похожие (тот же уровень/источник). |
| Гистограмма «±5 мин, корреляция» | Таймстемпов для корреляции по времени нет | Заменил на распределение выбранного уровня по окну предпросмотра. Честная метрика вместо вымышленной. |
| Live tail (`●`), trace_id, breadcrumb `acme/prod` | Нет стриминга, нет трейсинга | Не реализую. Breadcrumb показывает реальное: `studio / <источник> / <активные уровни>`. |

| Inspector → Контекст по `req_id` через сервисы | ~~один источник~~ → **cross-file** | «Контекст» собирает строки того же `req_id`/`trace_id` **по всем файлам сессии** — распределённый трейс. Строки помечены файлом-источником; клик переключает активный файл (трасса проходима шаг за шагом). Чистая функция `crossContext` в `core.js`, покрыта юнит-тестом. Ограничение: по окнам предпросмотра ≤1000/файл (наследует «предпросмотр — витрина»). |

**Вывод исследователя:** ~70% концепта ложилось на ядро честно сразу; мультифайловые
сессии и cross-file корреляция по `req_id` **теперь реализованы** (клиентская сессия,
backend не тронут). Остаются нереализованными live tail и корреляция по времени —
им нет надёжной опоры (стриминга и сквозных таймстемпов нет). Их «пустышкой» не рисую.

## 4. UX-дизайнер — что улучшилось и где компромиссы

**Плюсы 3-pane против старого 2-pane:**

- Запись больше не теряется в простыне `<pre>`: выбор строки → раскрытая
  структура справа. Это и есть «режим расследования».
- Поток — табличная сетка с цветокодом уровней и подсветкой выбранного, а не
  монотонный текст. Скан глазами быстрее.
- Поиск вынесен в топ-бар (⌘K), фильтры уровней — в постоянно видимый левый
  рельс со счётчиками. Меньше «где же это».
- **Навигация по трассе запроса.** Выбор хранится по идентичности записи, а не
  по индексу строки. Поэтому строки в «Контексте»/«Похожих» кликабельны: клик
  делает запись активной (подсветка + проскролл в потоке, переключение
  инспектора), а «Контекст» остаётся на том же `req_id` — можно идти по запросу
  шаг за шагом. Запись, скрытую фильтром потока, инспектор всё равно открывает с
  пометкой «скрыта фильтром». Поток листается стрелками ↑/↓.

**Компромиссы и риски (честно):**

- **Узость на малых экранах.** Сетка `268px / 1fr / 380px` < ~1100px душит
  центральную колонку. Нужен брейкпоинт: схлопывать правую панель в overlay.
  Сейчас не сделано — *known limitation*.
- **Контраст.** `--subtle #5d5577` на `--bg #0e0a14` — это ~3:1, ниже WCAG AA для
  мелкого текста. Унаследовано из концепта. Для прод-релиза поднять.
- **Плотность.** 11.5px моно — комфортно инженеру, тяжело для случайного
  пользователя. Это осознанный выбор аудитории (логи смотрят инженеры).

## 5. Консультант-аналитик — стоит ли мёрджить и куда дальше

**ROI редизайна:** высокий при низкой цене. Бэкенд не тронут → регрессионный риск
только во фронте, а фронт покрыт «глазами» + `observability.js` ловит JS-ошибки в
общий лог. Стоимость переключения для пользователя ≈ ноль (тот же контракт, тот же
запуск `./run.sh`).

**Рекомендация:** держать как **параллельный форк** (`-next`), не заменять
оригинал немедленно. Причина — оригинальный 2-pane проще и для разовой задачи
«распарсил → экспортировал» быстрее. 3-pane выигрывает в «расследовании», когда
строк много. Пусть оба поживут; победителя выберут по использованию.

**Сделано:** подсветка совпадений поиска (`<mark>`), `req_id` как сквозная ось
(кликабельный чип-пивот + токен `req:`), навигация ↑/↓, кликабельный контекст,
вынос чистого слоя в `core.js` + юнит-тесты.

**Мультифайловая сессия (реализовано).** Сессия живёт во фронте: `state.session =
{files[], activeId}`, каждый файл — независимый `ParseRequest` (backend не тронут).
Левая панель — дерево файлов (множественный drop, статус/формат/счётчик ошибок,
удаление, переключение активного). «Контекст» инспектора — cross-file трасса по
`req_id`. Экспорт: переключатель «активный файл / вся сессия» (сессия = N вызовов
`/api/export`, склейка в один `logzilla_session.json` `[{file, records}]`, zero-dep).
Настройки парсера глобальны для сессии. Чистый слой пополнен `crossContext` + юнит-тест.

**Приоритеты дорожной карты (по убыванию ценности):**

1. Адаптив < 1100px (закрывает реальный риск юзабилити; дерево файлов усугубило тесноту левой панели).
2. Контраст до WCAG AA (дешёвая правка палитры).
3. Виртуализация потока — сейчас рендерим ≤1000 DOM-строк; для роста окна нужна.
4. ~~Сессии/мультифайл~~ — **сделано**. Дальше — live tail / корреляция по времени,
   если бизнес подтвердит спрос. Это уже развитие ядра, а не редизайн.

## Запуск

```bash
cd logzilla-studio-next
./run.sh                 # → http://127.0.0.1:8765
# ядро берётся из ../logzilla3000-project (или $LOGZILLA3000_HOME)
```

Контракт, лимиты, безопасность (запрет `source.kind=file` по HTTP, path-traversal
guard, лимит тела 64 МБ) — **унаследованы из оригинала без изменений**.
