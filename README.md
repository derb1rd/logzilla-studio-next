## (оригинальный README ниже)

Сайд-проект: реализация концепта logZilla3000 как **локального веб-приложения** поверх
существующего ядра. Граница UI ↔ ядро вынесена на HTTP/JSON-контракт.

> **Отдельный проект.** Живёт ВНЕ стабильного репозитория ядра (`../logzilla3000-project/`),
> стабильный проект не изменяется. Ядро `logZilla3000` импортируется из соседней папки;
> путь переопределяется переменной окружения `LOGZILLA3000_HOME`.

## Зачем именно так (ответственный выбор стека)

Граница UI ↔ ядро вынесена на **HTTP/JSON-контракт** (а не на JS-мост PyWebView).
Причина — приоритет **отлаживаемости ИИ**: каждый баг = сохранённый JSON-запрос,
воспроизводимый одним `curl`, с полным DevTools браузера. Цель — богатый UI
поверх Python-ядра, с границей на самом отлаживаемом слое.

- **Ядро не тронуто** — пакет `logZilla3000` импортируется как есть.
- **Zero-dependency** — сервер на stdlib `http.server`. Нужен только `python3` (как и CLI ядра).
- **Framework-agnostic** — сервисный слой не знает про HTTP; позже легко заменить на FastAPI
  (auto-OpenAPI) или обернуть в PyWebView (webview → localhost) для desktop-вида.

## Запуск

```bash
cd logzilla-studio
./run.sh                 # → http://127.0.0.1:8765
# или: PORT=9000 ./run.sh
```

Открой `http://127.0.0.1:8765` в браузере → перетащи файл логов (или вставь текст) →
«Парсить» → смотри предпросмотр со статистикой → «Экспортировать» (JSON, опционально gzip).

## Структура

```
logzilla-studio/
  app/
    contract.py         # ParseRequest / ParseResult / ExportRequest (контракт границы)
    parse_service.py    # обёртка ядра: единый тип + метрики + диагностика
    export_service.py   # сериализация в JSON + опциональный gzip
    logging_setup.py    # структурные JSON-логи + run_id (наблюдаемость границы)
    server.py           # тонкий HTTP-слой (stdlib): /api/parse · /api/export · /api/client-log
  web/                  # фронтенд (dark theme):
    index.html
    styles.css
    app.js
    observability.js    # action log + перехват JS-ошибок + корреляция
  tests/                # контракт + сервисы + golden-фикстуры + smoke.py
  run.sh
```

## API (контракт)

```bash
# health
curl localhost:8765/api/health

# parse (inline)
curl -s localhost:8765/api/parse -H 'Content-Type: application/json' -d '{
  "source": {"kind":"inline","text":"2024-01-15 10:30:16 ERROR Database connection failed"},
  "options": {"log_levels":["ERROR","WARN"]},
  "preview": {"limit":1000,"offset":0}
}' | python3 -m json.tool

# export (повторно парсит тот же запрос → отдаёт JSON-файл; опц. gzip)
curl -s localhost:8765/api/export -H 'Content-Type: application/json' -d '{
  "parse_request": {"source":{"kind":"inline","text":"a,b\n1,2"}},
  "options": {"gzip": false}
}'
```

## Отладка GUI с ИИ

GUI — главный источник недетерминированных багов. Чтобы ИИ-агент мог их
воспроизводить, реализована наблюдаемость границы и фронта:

- **action log** — `web/observability.js` ведёт журнал действий пользователя
  (клики, выбор файла, парсинг, экспорт, поиск). Кнопка «⛏ action log» в футере
  скачивает его JSON — это и есть «сценарий воспроизведения», который отдают агенту.
- **Перехват JS-ошибок** — `window.onerror` и `unhandledrejection` уходят на
  `POST /api/client-log` и попадают в **общий структурный лог сервера** (закрывает
  слепую зону «JS-ошибки невидимы в Python-логе»).
- **Корреляция** — каждый запрос несёт `X-Correlation-Id` / `X-Session-Id`; сервер
  пишет их рядом с `run_id`. Так клиентское действие связывается с серверным прогоном.
- **PII-safe воспроизводимость** — сервер логирует `input_sha1` + `input_len`, а не
  содержимое лога.

Петля: воспроизвести по action log → по `correlation_id` найти серверный `run_id`
и `input_sha1` → проверить ядро на том же `ParseRequest` (право → баг в UI, нет → в ядре)
→ читать перехваченные JS-ошибки → чинить нужный слой.

```bash
# симулировать клиентскую JS-ошибку (как это делает фронт)
curl -s localhost:8765/api/client-log -H 'Content-Type: application/json' \
  -d '{"session_id":"s-x","kind":"js_error","message":"TypeError","recent_actions":[]}'
```

## Модель предпросмотра vs экспорт (решение заказчика)

- **Предпросмотр — полноценный рабочий просмотр с потолком окна 10 000 строк** (`PREVIEW_MAX`
  в `contract.py`). Раньше было 1000 («демонстрационная витрина»), но 3-pane инспектор с
  фильтрами/поиском/навигацией по трассе дорос до реального инструмента — листать стало
  удобно, потолок поднят до 10k. Если записей больше — в UI плашка «показаны первые 10000 из N».
  Выше упираемся в невиртуализированный рендер потока (виртуализация — пункт #3 дорожной
  карты в `REDESIGN.md`); до неё 10k — осознанный потолок (на верхней границе поток слегка
  подтормаживает).
- **Полный результат сверх окна — это парс в файл (Экспорт).** Экспорт перепарсивает полный
  `ParseRequest` и отдаёт весь набор (до `MAX_RECORDS`). Сюда (а не в предпросмотр)
  переносится работа по масштабу/стримингу очень больших файлов.

## Что НЕ вошло (вырезано / отложено)

- **История и Шаблоны — вырезаны из скоупа** (не требуются). Экспорт stateless:
